# SWAG 測試流程與標準（Test Process）

> 來源：Notion 測試流程文件、Bug Reporting 標準、FullTest Checklist、China QA 規範

---

## 一、版本發布節奏

### 前端（Web）
| 日程 | 動作 |
|------|------|
| **週二** | 前端版本上測試環境，QA 開始測試 |
| **週四** | 版本正式上線（Release） |

### 後端（BE）
- **無獨立測試環境**
- 所有後端變更**直接上正式環境**
- QA 以**回歸測試**為主，確保新版本不破壞現有功能

### Ramen / Moor App
- 依 Feature Branch 建置，以 `feature/china-app` 等分支為測試主體
- 不定期版本，依開發排程而定

## 二、測試環境

| 環境類型 | URL / 取得方式 |
|---------|--------------|
| Web 前端測試版 | `https://{v3-277}.app.swag.live`（版本號每次不同） |
| Web 前端正式版 | `https://swag.live/` |
| 前端測試版取得 | GitHub Repo → **View deployment**（左側 Deployments 列表） |

## 三、Bug 回報標準

### 3.1 Bug Report 必填欄位

```markdown
## [BUG] 標題（簡短描述問題現象）

### 嚴重程度
- [ ] P0 - 關鍵（系統癱瘓、資料遺失、金流異常）
- [ ] P1 - 嚴重（核心功能無法使用）
- [ ] P2 - 一般（功能異常但有 workaround）
- [ ] P3 - 輕微（UI 瑕疵、非核心功能）

### 重現步驟
1. 進入...
2. 點擊...
3. 觀察到...

### 預期行為
（說明正確的行為應該是什麼）

### 實際行為
（說明實際觀察到的錯誤行為）

### 環境脈絡
| 欄位 | 值 |
|------|---|
| 收集時間 | YYYY-MM-DD HH:MM:SS +08:00 |
| OS | （例：macOS 14.5 arm64） |
| 裝置類型 | desktop / mobile |
| 產品 | Web / Moor / Ramen / Sushi |
| 版本 | （App 版本號 or 前端 deploy SHA） |
| 網路 | WiFi / 有線 / 4G / 5G |
| VPN | 啟用 / 未啟用 |
| IP / 位置 | （例：114.x.x.x / Taipei, TW） |

### 附件
- [ ] 螢幕截圖 / 影片
- [ ] HAR 檔（網路流量記錄，Ramen 必附）
- [ ] Console Log
- [ ] Process ID & App 版本（Ramen/Moor）
```

### 3.2 Ramen Bug 必附資料
| 項目 | 說明 |
|------|------|
| 螢幕錄影 | 完整重現過程 |
| 網速截圖 | 測試當下的網速（Speedtest 截圖） |
| .har 檔 | Chrome DevTools → Network → Export HAR |
| Process ID | 裝置設定中取得 |
| App Version | 裝置設定中取得 |
| DevTool 流水圖 | Network 面板截圖（顯示 Timeline） |

## 四、FullTest 標準流程

> FullTest = 全功能回歸測試，通常在大版本上線前執行。

### 4.1 觸發條件
- 前端大版本發布前
- Moor 有重大功能更新時，Flutter Web 也需同步 FullTest
- Payment SDK 或金流商有更新

### 4.2 FullTest 核心檢查清單

#### 直播功能
- [ ] 主播開播（一般、Show、一對一、一對多）
- [ ] 用戶進入直播間（劇院模式、簡約模式）
- [ ] 打賞（所有類型禮物）
- [ ] 送指令
- [ ] 一對一流程（邀請 → 預扣 → 計費 → 退款）
- [ ] Show 流程（早鳥票 → 達標 → 開始 → 結束 → 收益結算）
- [ ] 封鎖/取消封鎖

#### 影片功能
- [ ] 上傳長影片（≥3 分鐘）
- [ ] 上傳短影音
- [ ] 發布限時動態（3–30 秒）
- [ ] 購買後進入解鎖項目（永久保存）
- [ ] 影片評論與回覆

#### 金流功能
- [ ] **信用卡**儲值（核心測試項目）
- [ ] **AFTEE** 先享後付儲值（核心測試項目）
- [ ] 鑽石扣除與 Log 記錄
- [ ] 退款/沖正記錄

> ⚠️ **支付寶、轉帳、虛擬幣**：FullTest **不需測試**

#### 帳號功能
- [ ] 修改電話號碼（不可刪除）
- [ ] 修改電子信箱（不可刪除）
- [ ] VIP 等級顯示
- [ ] 封鎖清單管理

### 4.3 中國 App（Ramen）FullTest 額外項目
- [ ] 首頁載入時間 < 10 秒
- [ ] 直播間進入速度
- [ ] API 加密正常（x-encrypted-algo header 存在）
- [ ] SWAG 字樣已隱藏（China 版要求）
- [ ] 6 城市撥測通過（詳見 China QA 規範）

## 五、China QA 規範

### 5.1 城市排程測試

每週固定對以下 6 個城市進行撥測：

| 城市 | 工具 | 標準 |
|------|------|------|
| 北京 | BOCE 波測 | 載入 < 10 秒 |
| 上海 | BOCE 波測 | 載入 < 10 秒 |
| 廣州 | BOCE 波測 | 載入 < 10 秒 |
| 深圳 | BOCE 波測 | 載入 < 10 秒 |
| 成都 | BOCE 波測 | 載入 < 10 秒 |
| 武漢 | BOCE 波測 | 載入 < 10 秒 |

**BOCE 登入**：admin@swag.live / Swag5566

### 5.2 域名監控

每日必查：
- **ICP 備案**：工業和信息化部政務服務平台 → 備案查詢
- **BOCE 撥測**：確認目標域名在中國可連通

### 5.3 China 版 App 特殊要求
| 要求 | 說明 |
|------|------|
| 隱藏 SWAG 字樣 | App 內不可出現 `swag` 品牌字樣 |
| 對應分支 | `feature/china-app` |
| API 加密 | Response Body 加密，Header `x-encrypted-algo` 標明演算法 |

## 六、測試標準 SLA

| 嚴重度 | 定義 | 預期回應時間 |
|-------|------|------------|
| **P0** | 金流異常、系統癱瘓、資料遺失 | 立即回報，當日修復 |
| **P1** | 核心功能無法使用（直播、打賞、支付） | 24 小時內修復 |
| **P2** | 功能異常但有替代方案 | 1 週內排入修復 |
| **P3** | UI 瑕疵、文字錯誤、非核心功能 | 下次版本處理 |

## 七、測試案例命名規範

```
TC-{產品}-{功能模組}-{簡述}

範例：
TC-WEB-PAYMENT-信用卡儲值成功
TC-MOOR-1ON1-超過預扣時間關播
TC-RAMEN-LOAD-北京城市撥測10秒
TC-SUSHI-SWITCH-首頁切換直播間
```

## 八、QA 與 Dev 協作規範

| 情境 | 處理方式 |
|------|---------|
| Dev 說「是 Known Issue」 | 確認是否已記錄於知識庫或 Issue，並標記為「Known」 |
| 無法重現 | 附上環境脈絡，請 Dev 確認是否環境限定 |
| BE 直接上線導致異常 | 立即 P0 回報，確認回滾可能性 |
| Ramen 加密問題 | 透過前端行為或後台資料比對（不能直讀 Network Response） |
