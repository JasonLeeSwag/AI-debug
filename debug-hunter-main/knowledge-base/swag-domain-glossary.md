## 一、SWAG 業務核心術語

| 術語 | 說明 | 注意事項 |
|------|------|---------|
| 點數 / 分 | SWAG 平台的虛擬貨幣單位 | 又稱「swag 幣」、「點」，購買後不可退；與法幣之間有固定兌換比例；**不等同於新台幣** |
| 買分 | 用法幣（台幣/人民幣等）購買 SWAG 點數的行為 | 需支付平台手續費；透過第三方金流（綠界/支付寶/微信支付）完成；入帳後不可退款 |
| 打賞 | 用戶將點數作為禮物送給正在直播的主播 | 即時扣款，不可撤銷；主播立即看到動畫效果；扣款後由平台計算分潤 |
| 禮物 | 打賞的另一種形式，以虛擬禮物（玫瑰、跑車等）呈現 | 不同禮物對應不同點數面額與動畫效果；禮物面額即打賞金額 |
| 訂閱 | 用戶以點數按月訂閱特定主播，可存取訂閱限定內容 | 月費制；訂閱期間到期若點數不足則自動取消；訂閱不等於一次性解鎖 |
| 訂閱限定內容 | 主播設定為僅供訂閱用戶可見的直播或媒體檔案 | 非訂閱用戶呼叫 API 應收到 403；訂閱到期後立即無法存取 |
| 主播 | SWAG 平台上的內容創作者（直播主） | 有不同等級（V1/V2/V3/V4 等）；等級影響分潤比例；需經過平台審核 |
| 分潤 | 平台將打賞收益依比例分配給主播的收益 | 通常是打賞金額的固定百分比（依主播等級 50%~70%）；**「分潤」≠「全額」** |
| 分潤比例 | 主播從每筆打賞中獲得的百分比 | 依主播等級而異；計算時必須使用 Decimal 以避免浮點誤差 |
| 出金 | 主播將累積的分潤（點數或法幣）提領至銀行帳戶 | 需通過身分驗證（KYC）；有最低出金門檻；需區分「主播出金」與「博弈出金」（同詞不同義） |
| 龍虎鬥 | SWAG 博弈遊戲之一，比較龍方與虎方牌面大小 | 類似撲克牌比大小；分為「龍勝」「虎勝」「和局」；平台有莊家邊（house edge） |
| 百家樂 | SWAG 博弈遊戲之一，莊閒對決 | 分為「莊贏」「閒贏」「和局」；莊家贏收取 5% 佣金（抽水）；和局賠率 8:1 |
| 局 / Round / 期 | 博弈遊戲的一輪（從下注到開獎） | 三個詞等義；局號是本輪的唯一識別碼 |
| 賠率 | 博弈贏錢的倍數 | **需考慮平台抽水後的淨賠率**；賠率 1:1 代表贏得等額押注 |
| 押注 | 博弈遊戲中下注的行為 | 有最小押注與最大押注限制；同一局可多次押注不同選項 |
| 退水 | 博弈中未贏得的押注退還部分（依促銷活動） | 與「出金」不同；退水是促銷返還，出金是提款到銀行 |
| 莊家邊 | 平台在博弈中的數學期望優勢 | 百家樂：莊家贏抽 5%；龍虎鬥：各選項約 1%~3%；確保平台長期盈利 |
| 房間 | 一個主播的直播頻道（每位主播有唯一房間） | 用戶進入房間才能觀看直播並互動；房間有觀看人數統計 |
| 即時串流 | Live streaming，主播透過 RTMP 推流，用戶透過 HLS/WebRTC 收看 | 延遲約 3~30 秒（依協議）；需區分「直播中」與「重播/VOD」 |
| 私密聊天 | 用戶與主播的一對一付費視訊互動 | 按分鐘計費；比一般直播打賞費率更高；技術上使用 WebRTC P2P |
| KYC | Know Your Customer，身分驗證流程 | 主播出金前必須完成；需身分證件或護照驗證；台灣居民另可用金融卡驗證 |

## 二、技術術語（SWAG 語境說明）

| 術語 | SWAG 語境說明 | 注意事項 |
|------|-------------|---------|
| 回調 / Callback | 第三方支付平台（綠界/支付寶/微信支付）在用戶完成支付後，向 SWAG 伺服器發送的 HTTP POST 通知 | 回調需驗證簽章；需做冪等處理（同一筆可能重複發送）；回調成功才入帳點數 |
| 訂單號 | SWAG 系統內部產生的唯一訂單識別碼 | **不等同於第三方平台的「交易號」**；是 SWAG 自己的 ID |
| 交易號 | 第三方支付平台（綠界/支付寶/微信支付）產生的訂單 ID | 每家叫法不同（見第四節）；是做冪等鍵的核心欄位 |
| 流水號 | SWAG 內部每一筆操作記錄的唯一序號 | 廣義上包含：點數流水、支付流水；用於對帳與追蹤 |
| 點數流水 | 用戶或主播帳戶點數增減的歷史記錄 | 每筆打賞、訂閱、退款都產生一條流水；流水只增不改（不可刪除） |
| 冪等鍵 | 用於防止重複請求的唯一識別碼 | 支付回調用「交易號」；打賞用「request_id（UUID）」；必須存入 Redis 或 DB |
| 即時串流 | 技術上指 RTMP/HLS/WebRTC 直播協議 | RTMP 用於推流，HLS 用於拉流（延遲高），WebRTC 用於低延遲互動 |
| 沙箱 | 第三方支付平台的測試模式，不產生真實金流 | 各平台沙箱有獨立的測試帳號和測試卡號 |
| 分潤帳本 | 記錄主播累積未出金分潤的帳戶 | 打賞發生時分潤寫入帳本；出金時從帳本扣除 |
| 房間 ID | 直播房間的唯一識別碼 | 等同於主播的直播頻道 ID；WebSocket 連線以房間 ID 分群組 |
| 推流金鑰 | 主播使用 OBS 推流時的認證金鑰 | 需保密；洩漏後任何人可冒用主播身份推流 |

## 三、禁止混淆的術語對

> 以下術語組合極易產生誤解，AI 代理人與 QA 工程師在溝通時**必須明確區分**。

| 容易混淆 A | 容易混淆 B | 區別說明 |
|-----------|-----------|---------|
| 點數餘額（虛擬貨幣） | 法幣金額（台幣/人民幣） | 點數是平台幣，不可直接用於法幣場景；換算率由平台決定，不固定等於 1:1 |
| 出金（主播提款） | 出金（博弈出金） | 同一詞但語境不同：主播出金=分潤提領至銀行；博弈出金=贏錢後提領點數。**需依業務上下文判斷** |
| 退水（博弈促銷退還） | 退款（金流退款） | 退水是博弈活動中返還押注比例；退款是法幣/點數退回用戶帳戶；**退水不等於退款** |
| 分潤（主播收益分配） | 分配（通用詞） | 「分潤」專指主播從打賞中獲得的收益比例；不要將「分潤」用於其他場景 |
| 訂單號（SWAG 內部 ID） | 交易號（第三方平台 ID） | 兩者都是唯一 ID，但來源不同；對帳時需同時保存兩個 ID；冪等鍵通常用交易號 |
| 訂閱（月費制，點數扣款） | 付費解鎖（單次購買特定內容） | 訂閱是持續性服務；付費解鎖是一次性購買；到期邏輯不同 |
| 打賞（即時送點數給主播） | 付費看直播（訂閱或解鎖才能看） | 打賞是自願給主播的禮物；付費直播是存取費；打賞不等於訂閱 |
| 主播等級（V1/V2/V3...） | 用戶等級（會員等級） | 主播等級影響分潤比例；用戶等級影響享有的優惠；兩套等級系統獨立 |
| 房間（直播頻道） | 頻道（Channel，同義詞） | 在 SWAG 語境中通常說「房間」；部分文件說「頻道」，兩者相同 |
| 快取（Cache，Redis） | 資料庫（DB，持久化） | 訂閱狀態快取≠資料庫紀錄；快取可能過期；以 DB 為準 |

## 四、支付平台術語對照

> SWAG 與各支付平台的欄位名稱不同，開發/測試時需對應正確欄位。

| SWAG 內部稱法 | 綠界 ECPay | 支付寶（Alipay） | 微信支付（WeChat Pay） | 說明 |
|-------------|-----------|----------------|----------------------|------|
| 訂單 ID（SWAG） | MerchantTradeNo | out_trade_no | out_trade_no | 商家方（SWAG）產生的唯一訂單號 |
| 支付金額 | TotalAmount（台幣，元） | total_amount（人民幣，元，最多2位小數） | total_fee（人民幣，**分**，整數） | 注意微信支付單位是「分」，不是「元」！|
| 交易號（第三方） | TradeNo | trade_no | transaction_id | 第三方平台產生的交易流水號，用作冪等鍵 |
| 支付狀態 | TradeStatus（字串） | trade_status（字串） | result_code（字串） | 各平台狀態碼不同，見下表 |
| 支付成功狀態值 | "Paid" | "TRADE_SUCCESS" | "SUCCESS" | 比對時需注意大小寫 |
| 回調通知 URL | PaymentInfoURL（非同步）/ ReturnURL（同步） | notify_url | notify_url | SWAG 需公網可訪問的接收端點 |
| 退款訂單號 | MerchantTradeNo（另建一筆） | out_refund_no | out_refund_no | SWAG 產生的退款訂單 ID |
| 金流手續費 | 依費率合約 | 費率依商家等級 | 費率依商家等級 | 通常由 SWAG 承擔，不向用戶收取 |

**支付金額單位陷阱**（高頻 Bug 來源）：

```python
# 危險：微信支付金額單位是「分」，不是「元」
# 錯誤：100 台幣 ≠ 100（人民幣元）≠ 100（微信分）
# 正確換算流程：
# 1. 台幣換人民幣（依匯率）
# 2. 人民幣元 × 100 = 微信支付的 total_fee（分）

def amount_to_wechat_fee(amount_cny_yuan: Decimal) -> int:
    """人民幣元轉微信支付分（整數）"""
    fee_in_fen = amount_cny_yuan * Decimal('100')
    return int(fee_in_fen)  # 微信支付只接受整數

# 綠界：台幣，整數元
def amount_to_ecpay(amount_twd: Decimal) -> int:
    """台幣轉綠界金額（整數元）"""
    return int(amount_twd.quantize(Decimal('1')))  # 無小數
```

## 五、測試術語（QA 內部專用）

| 術語 | 說明 | 使用場景 |
|------|------|---------|
| MRS | Minimal Reproducible Scenario，最小復現情境 | 回報 Bug 時提供，確保問題可被快速重現 |
| E2E | End-to-End，端到端測試 | 模擬真實用戶操作完整流程，從 UI 到 DB |
| 沙箱 | Sandbox，各支付平台的測試環境 | 測試時必須使用沙箱，絕不用正式環境 |
| 煙霧測試 | Smoke Test，快速驗證主要功能正常 | 每次部署後 5 分鐘內跑完，確認基本功能未壞 |
| 回歸測試 | Regression Test，確保修改不破壞原有功能 | 每次 PR 合併前必跑；包含所有歷史 Bug 的修復驗證 |
| 負向測試 | Negative Test，測試不合法輸入或邊界情況的處理 | 如：餘額不足時打賞、過期 Token 呼叫 API |
| 邊界測試 | Boundary Test，測試臨界值（最大/最小/剛好等於） | 如：最低出金門檻、最大押注限制 |
| 探索性測試 | Exploratory Testing，無腳本的自由探索測試 | 新功能上線前由 QA 人工探索，發現腳本外的 Bug |
| Flaky Test | 不穩定測試，相同條件下有時通過有時失敗 | 使用固定 Sleep 是常見原因；需改用條件等待修復 |
| 黃金路徑 | Golden Path，最典型的正常使用流程 | 如：用戶購買點數 → 打賞主播 → 主播出金 |
| 測試隔離 | 每個測試彼此獨立，不共享狀態 | 每個 Suite 建立獨立測試帳號，Teardown 後清理 |
| 冒煙測試 | 同「煙霧測試」（台灣 QA 社群常見別名） | 見煙霧測試說明 |
| P0 / P1 / P2 | Bug 嚴重程度分級 | P0：阻斷業務流程（如重複扣款）；P1：主要功能異常；P2：次要功能異常 |

## 六、台灣正體中文技術用語規範

> SWAG 業務文件與程式碼注釋應使用台灣正體中文。以下列出常見的兩岸用語差異。

| 大陸簡體中文用語 | 台灣正體中文對應 | SWAG 業務用語 |
|--------------|--------------|-------------|
| 充值 | 儲值 / 購買點數 | 在 SWAG 語境中統一說「買分」或「購買點數」 |
| 提现 | 提款 / 出金 | 主播場景用「出金」；博弈場景也用「出金」（需看上下文）|
| 礼物 | 禮物 / 虛擬禮物 | SWAG 文件統一用「禮物」 |
| 打赏 | 打賞 | 兩岸同詞但注意繁簡字體 |
| 直播 | 直播 | 相同 |
| 游戏 | 遊戲 | SWAG 博弈文件用「遊戲」 |
| 庄家 | 莊家 | 百家樂術語用「莊家」 |
| 闲家 | 閒家 | 百家樂術語用「閒家」 |
| 流水号 | 流水號 | SWAG 術語用「流水號」或「交易流水」 |
| 回调 | 回調 | 支付通知機制，SWAG 統一用「回調」 |
| 接口 | 介面 / API | 技術文件用「API」，避免「接口」 |
| 数据库 | 資料庫 | SWAG 技術文件用「資料庫」 |
| 缓存 | 快取 | SWAG 技術文件用「快取」（Redis 快取） |
| 签名 / 签章 | 簽章 | 支付驗證場景用「簽章」 |
| 幂等 | 冪等 | SWAG 技術文件用「冪等」（冪等性、冪等鍵） |
| 加密 | 加密 | 相同 |
| 解密 | 解密 | 相同 |
| 中间件 | 中介軟體 | FastAPI middleware 文件用「中介軟體」 |
| 异步 | 非同步 / 異步 | SWAG 技術文件用「非同步」（async/await） |
| 并发 | 並發 | 相同（注意繁簡） |

## 七、業務流程術語速查（QA 測試情境對應）

| 測試情境 | 涉及術語 | 常見誤解 |
|---------|---------|---------|
| 用戶購買點數流程 | 買分、法幣、訂單號、交易號、回調 | 點數不等於台幣；微信支付金額單位是「分」 |
| 打賞主播流程 | 打賞、禮物、分潤、點數流水、冪等鍵 | 分潤≠全額；需驗算分潤比例正確性 |
| 訂閱主播流程 | 訂閱、到期、自動續訂、訂閱限定內容 | 訂閱≠付費解鎖；到期後須即時無法存取 |
| 主播出金流程 | 出金、KYC、分潤帳本、最低出金門檻 | 出金需先驗 KYC；出金失敗需回滾分潤帳本 |
| 博弈遊戲流程 | 局、押注、賠率、莊家邊、抽水、退水 | 退水≠退款；莊家贏有抽水（5%） |
| 支付回調處理 | 交易號、冪等鍵、回調、支付狀態碼 | 回調可能重複發送；需冪等處理 |
| 年齡驗證 | KYC、18+驗證、NSFW | 前端驗證≠後端驗證；後端必須強制驗證 |
